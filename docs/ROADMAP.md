# ROADMAP

Дорожная карта продукта. Два трека делят общее ядро (`jobsearch/`):
личный пайплайн (рабочий) и многопользовательское приложение на той же логике.

> Статусы: ✅ сделано · 🚧 в работе · ⬜ запланировано

## Принятые решения

### Стек
- **Supabase** — Postgres + Auth + Storage + RLS, **EU-регион**. Managed-сервис; свой
  auth/хранилище не пишем.
- **FastAPI** — backend-mediated: фронт ходит **ТОЛЬКО** в FastAPI, никогда напрямую в
  Supabase с привилегиями. `service_role`-ключ живёт **только на бэке**.
- **Next.js** — фронт (B2C-кабинет соискателя).
- Мобайл и B2B — отложены в **Фазу 3**.

### Безопасность
- **Изоляция арендаторов через Postgres RLS** как defense-in-depth: даже при баге в коде
  приложения БД не отдаёт чужие строки. Ложится напрямую на интерфейс `UserState` (в нём
  по контракту нет кросс-юзерных чтений — каждый метод скоупится `user_id`).
- **Managed auth** (Supabase) — свой не пишем.
- **Секреты только в окружении бэкенда**: ключ Anthropic и `service_role` не попадают во
  фронт-бандл и не пишутся в логи (инвариант уже в `scoring.AnthropicClient` — ключ из env,
  маскирован в repr).
- **EU-резидентность под GDPR** + право на удаление: каскадное стирание данных юзера; в
  `UserState` появляется метод `delete` (полное удаление персонального состояния).
- **Шифрование at-rest и бэкапы** — на стороне платформы (Supabase из коробки).
- **Квоты / rate-limit на юзера** — одновременно контроль стоимости LLM и анти-абьюз.
- **Недоверенные описания вакансий** идут только в user-позицию LLM-промпта (инвариант уже
  в `scoring.py`; есть тест-регрессия).

### Продукт
- **B2C-цикл сначала** (соискатель): регистрация → мастер-CV → ключевики/период →
  подогнанные CV + сопроводительное + итог по вакансии.
- **B2B** (агентства/работодатели) и платежи — в **Фазе 3**.

## Фаза 0 — чистое ядро ✅
Ядро вынесено из CLI без изменения поведения скрейпинга/скоринга/тюнинга.
- `scrape(params, config) -> list[Job]` — state-pure: настройки в параметрах, seen/кэш не
  внутри (живут над скрапером, см. ТЗ §4).
- `score_fit(job, short_profile, config, client) -> PreScore` — дешёвый Haiku-предфильтр.
- `analyze(job, cv_text, config, client) -> MatchResult` — полный Sonnet-тюнинг; CV приходит
  аргументом; `LLMClient` инжектируется (тесты гоняют скоринг с фейковым клиентом, без сети).
- `build_package(job, res, cv_text) -> Package` — резюме (docx-байты) + cover letter +
  ATS-отчёт **в памяти**, ничего не пишет на диск.
- Состояние за интерфейсами `JobStore` / `UserState`; CLI (`finder` / `pipeline`) — тонкие
  обёртки, вся I/O на краю. Тесты зелёные.

## Фаза 1 — структура и инфраструктура ✅
- Пакетный layout `jobsearch/`, `scripts/`, `deploy/`, `tests/`, `pyproject.toml`.
- Тесты ядра, установка через `pip install -e .`.

## Фаза 2a — walking skeleton ⬜
Один внешний юзер, весь цикл от регистрации до скачивания комплекта, задеплоено.
- **Supabase-схема:** `cvs` (`user_id`), `search_params` (`user_id`), `jobs` (общий пул,
  `dedup_key` UNIQUE), `matches` (`user_id`, `job_id`, `fit`, `status`, ссылки на артефакты).
  **RLS** на всех пер-юзерных таблицах.
- **DB-реализации существующих Protocol'ов:** `SupabaseJobStore(JobStore)` и
  `SupabaseUserState(UserState)` — drop-in замена `FlatFileJobStore` / `FlatFileUserState`
  без правок вызовов ядра. `Package.cv_docx` → **Supabase Storage**, ссылки в `matches`.
- **FastAPI поверх ядра:**
  - загрузка + LLM-парсинг CV в схему `master_cv.md`;
  - сохранение `SearchParams`;
  - эндпоинт **Run**: `scrape` → запись в `JobStore` с дедупом → по необработанным:
    `score_fit`-гейт → `analyze` → `build_package` → Storage;
  - список матчей; скачивание комплекта.
- **Минимальный Next.js:** логин, CV, форма ключевиков/периода, Run, список результатов.
- **Деплой:** бэк на Railway/Render (EU), фронт на Vercel. Обработка синхронная; скрап по
  запросу, но с записью в **общий пул**; квота = жёсткий потолок на юзера.

## Фаза 2b — автоматизация и устойчивость ⬜
- **Очередь + воркер** (n8n или Arq/Celery) — асинхронный Run вместо синхронного.
- **Серверное расписание per-user** — замена локального LaunchAgent.
- **Централизованный плановый скрапер** по широким категориям + кэш `query_hash` + TTL через
  `JobStore` (анти-бан; отложено в ТЗ §4 — кэш живёт над `scrape`, не внутри).
- **Квоты / бюджеты на юзера.**
- **Дашборд воронки откликов**; уведомления (email / Telegram).

## Фаза 3 — масштаб, качество, расширение ⬜

- **Семантический матчинг** (эмбеддинги) для преранжирования пула до дорогого LLM-шага.
- **Расширение источников** через легальные API.
- **Мобайл** (Expo).
- **B2B и платежи.**
